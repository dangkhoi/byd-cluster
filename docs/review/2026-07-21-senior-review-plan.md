**W1-1 · Phantom turn instruction latched on the physical cluster for 3 minutes** (RC-8, from SAF-01)
*Evidence: PROVEN.* `AutotestActivity.kt:576-587` calls `ClusterBroadcaster.emit(distance="250 m", road="Nguyễn Huệ", …)` then `BydHal.writeNavFrame(ctx, 2, 300, "InProc")`. `ClusterBroadcaster.emit → scheduleHeartbeat` (ClusterBroadcaster.kt:78, 169-184) re-sends every `HEARTBEAT_MS=400` until `STALE_MS=180000` (:21-22). The default branch never calls `ClusterBroadcaster.stop` or `BydHal.clearNavFrame`. One documented `am start` leaves a fabricated turn arrow + street name on the gauge the driver reads, surviving the laptop being unplugged.

- **Generic fix:** *every path that writes to the cluster HAL owns a teardown in a `finally`.* Wrap the default autotest branch: `try { … } finally { ClusterBroadcaster.stop(applicationContext); BydHal.clearNavFrame(applicationContext) }`. Do not special-case the autotest — audit for any other `writeNavFrame`/`emit` caller without a paired teardown and apply the same shape.
- **Files:** `modules/AutotestActivity.kt` (:576-600), `ClusterBroadcaster.kt` (confirm `stop()` cancels the heartbeat runnable), `modules/hal/BydHal.kt` (`clearNavFrame` exists — verify it is called at least once, per §8).
- **Test:** unit — a fake broadcaster records `emit`/`stop`; assert the default autotest branch ends with `stop` called exactly once even when `writeNavFrame` throws.
- **Blast radius if wrong:** low. Worst case the autotest report loses its "look at the cluster" window — mitigate by sleeping 5 s before teardown so the tester still sees the frame.

**W1-2 · `selfTest()` tears down a live mock mid-tunnel; orphaned test provider kills fleet GPS** (RC-8 + RC-2, from SAF-01 + SAF-10)
*Evidence: PROVEN for the contract violation (MockLocModule.kt:32-35, DeadReckonModule.kt:38 both call `MockLoc.start`+`stop`); LIKELY for the orphan (window is real per CLAUDE.md §3 — `addTestProvider` evicts the real provider from `mProviders` and `native_stop()`s the engine; AOSP 10 `LocationManagerService` has no death-recipient cleanup for test providers).*
Two halves of one hole: the probe mutates, and there is no sweep for what it leaves behind. Current undo paths (`DeadReckonService.onDestroy:244` posted to a worker Handler, the `onCreate:183` sweep, `RebindReceiver:37/:51` on BOOT/PACKAGE_REPLACED) are all in-process and all miss force-stop and native crash.

- **Generic fix, two parts, both generic:**
  1. **Contract:** state in `ClusterModule` KDoc that `selfTest()` MUST NOT mutate system state. Rewrite both offenders to *observe*: read `Settings.Secure.getString(cr, "mock_location")` / appops instead of calling `addTestProvider`.
  2. **Durable sweep on the alarm that already exists:** `MockLoc.start` writes `K_MOCK_ON=true` to prefs with `.commit()` **before** `addTestProvider`; `stop` clears it after `removeTestProvider`. Add `ACTION_WATCHDOG -> { rebind(ctx); MockLoc.sweepIfOrphaned(ctx) }` in `RebindReceiver.onReceive`, where sweep = `if (marker && !(DeadReckonState.running && active)) stop(ctx)`. Manifest-declared receiver ⇒ fires on a cold process, which is exactly the case the three current hooks miss.
- **Files:** `modules/mockloc/MockLoc.kt` (:32-46, :57), `modules/mockloc/MockLocModule.kt` (:31-35), `modules/deadreckon/DeadReckonModule.kt` (:34-38), `RebindReceiver.kt` (:33-52, :102-118), `ClusterModule` interface KDoc.
- **Test:** `verify(never())` that `ModuleRegistry.MODULES.map { it.selfTest(ctx) }` calls `addTestProvider`; `InOrder` assert marker-write precedes `addTestProvider`; sweep with marker=true + `running=false` calls `removeTestProvider` once and clears the marker; with `running=true` calls nothing.
- **Blast radius if wrong:** **HIGH — this is the one to be most careful with.** A sweep that misfires while DR is legitimately mocking removes the provider mid-tunnel; a sweep that reads the marker wrong could remove a provider owned by a *different* app. Gate strictly on `DeadReckonState.running` and only ever call `removeTestProvider` for our own provider name. Ship this behind a 2-consecutive-negatives debounce.
- **Closing observation to promote LIKELY→PROVEN:** with DR mocking, `adb shell am force-stop com.byd.clusternav`, then `dumpsys location | grep -i -A3 'mock\|test provider'`.

**W1-3 · COLD_SEED stationary gate fails open where the speed HAL is unreadable** (RC-3, from SAF-08)
*Evidence: LIKELY (mechanism PROVEN, vehicle population unmeasured).* `SpeedProvider.mps()` returns 0.0 forever when the HAL is absent (SpeedProvider.kt:21-30 — four `return lastGoodMps` paths, `lastGoodMps` init 0.0 at :18). `DeadReckonService.kt:293` gates cold-seed on `speed < 2.0` whose stated purpose is "only when parked". The other two conditions are satisfied by a real tunnel after a restart (`!hasFix` — `MY_PACKAGE_REPLACED` restarts DR at RebindReceiver.kt:47-52; `sats < COLD_SEED_MAX_SATS=8`). Amplified deliberately: `coldSeedStationary=true` freezes the position (:310) and `:365` exempts COLD_SEED from `MAX_DR_MS`. Result: the whole vehicle's GPS pinned to a coordinate up to 7 days old, released only by a real fix better than 75 m — in a tunnel, never.

- **Generic fix:** make "unknown" a type, and make every safety gate demand a *positive* observation.
  `fun mpsOrNull(): Double?` returns null on every failure path; `fun mps() = mpsOrNull() ?: lastGoodMps` stays for the display/interpolation path where degrading is correct. Gate becomes `SpeedProvider.mpsOrNull()?.let { it < 2.0 } == true` — unreadable ⇒ not stationary ⇒ no cold-seed. Apply the same shape to the DR-entry heading gate.
  Second, independent belt: cap seed age for cold-seed at ~30 min. The 7-day store cap serves display/diagnostics only; a coordinate broadcast as the vehicle's *live* position needs a far tighter bound.
- **Files:** `SpeedProvider.kt` (:20-30), `modules/deadreckon/DeadReckonService.kt` (:293-311, :429-431).
- **Test:** DR state-machine unit test with an injected speed source — source reporting "unreadable" + `!hasFix` + `savedValid` + `sats<8` must NOT enter COLD_SEED; source reporting 0.4 m/s must. Second case: cold-seed refused when `savedAgeMs` exceeds the fresh cap.
- **Blast radius if wrong:** medium. Over-tightening means cold-seed never fires on some cars → tunnel entry from a cold start shows no position instead of a stale one. That is the correct failure direction, and it is what the pre-cold-seed builds did.
- **Closing observation:** on an SL6/Han, run the autotest Speed live-read (`AutotestActivity.kt:548-550`) and confirm whether `getCurrentSpeed` returns a number or null/sentinel.

**W1-4 · Unattended boot cast, no motion gate, R3 force-stop enabled by a silent migration** (RC-4 + RC-3, from SAF-09 + ARCH-07)
*Evidence: PROVEN.* `autoCastOnBoot` (ClusterCast.kt:868-878) sleeps `BOOT_CAST_DELAY_MS=25000` on a bare `Thread`, then runs the full `cast()` — including `[30,16,35]` which reconfigures the physical cluster, and including rung R3 `am force-stop`. Nothing checks gear, speed, or motion. 25 s after boot is when the car is being reversed out. Compounding: `applyDefaultModes` (:191-200) runs once per version and does `keepSessionApps - PKG_ANDROID_AUTO - PKG_CARPLAY` — a migration that *removes* the user's protection against R3 for the two apps where losing the session costs an active call route or projection.

- **Generic fix, three parts:**
  1. **Never widen a permission in a migration.** Delete `applyDefaultModes` (:191-200) and both package literals (:121-122). A default may only move conservative. (Product note: this touches ISSUES.md item 3 — see §4 conflict C-6.)
  2. **Unattended casts may not be destructive.** `cast(app, pkg, log, allowDestructive = false)` on the auto-cast path; `placeLadder` refuses to emit `am force-stop` when false. R3 stays available to a user who is looking at the screen and pressed the button.
  3. **Positive stationary observation before an unattended cluster mutation:** `if (SpeedProvider.mpsOrNull()?.let { it < 0.5 } != true) { log("bỏ qua tự chiếu — xe đang chạy hoặc chưa đọc được tốc độ"); return }`, re-checked immediately before `cast()` dispatches.
  4. Replace the sleep-based ordering with an explicit chain: `reconcileOnStart` takes a completion callback and posts `autoCastOnBoot` from it (today `RebindReceiver.kt:81` queues reconcile onto `vdExec` while `:44` starts an independent thread — the ordering is assumed, not expressed).
- **Files:** `ClusterCast.kt` (:121-122, :191-200, :579-586, :868-878), `RebindReceiver.kt` (:44-52, :81).
- **Test:** `loadPrefs` upgrade from a blob containing `keepSession=com.byd.carplay.ui` must NOT remove that entry; `placeLadder(allowDestructive=false)` never emits `am force-stop` against a fake `sh` recorder; `autoCastOnBoot` with an unreadable speed source does not call `cast()`.
- **Blast radius if wrong:** medium-high. Part 3 depends entirely on W1-3 — if `mpsOrNull` is not landed first, `!= true` on a 0.0-forever source *blocks* auto-cast on every affected car, which is a visible feature regression (but a safe one). **Hard ordering: W1-3 before W1-4.**

**W1-5 · `busy` latch + untimed dadb means one hung cast permanently blocks TẮT** (SAF-07, own cause)
*Evidence: LIKELY.* `busy.compareAndSet` at ClusterCast.kt:260, released only in the executor `finally` at :418; `stop()` refuses to run while busy at :423. All 12 `Dadb.create("localhost", 5555, …)` sites (ClusterCast.kt:267, :428, :714, :773, :802, :885; MockLoc.kt:127; …) pass no connect or read timeout, and the cast body runs inside `adb.use { }` with unbounded blocking reads plus 2.5 s/3 s/1.5 s sleeps. adbd wedging is not hypothetical — it is the same adbd affected when CarPlay/AA attach and detach WiFi, which is exactly when casting is used. End state: cluster projected, app refuses to tear it down, `vdExec` never drains so `applyScaleLive` and `reconcileOnStart` stop working too.

- **Generic fix:** bound the operation, not the button.
  `private fun <T> onVd(name: String, timeoutMs: Long = 60_000, body: () -> T)` submits to `vdExec`, keeps the `Future`, `f.get(timeoutMs, MILLISECONDS)`, on timeout `f.cancel(true)` + release `busy` on the *submitting* side. Give the socket a deadline: dadb accepts a pre-connected socket — `Socket().apply { connect(InetSocketAddress("localhost",5555), 4000); soTimeout = 15000 }`. And make `stop()` reachable under duress: once the latch has been held past the timeout, `stop()` proceeds anyway (it is idempotent and its guards are ground-truth based).
- **Files:** `ClusterCast.kt` (:255-270, :418-433), a shared `AdbGateway` for socket construction (repo global §4.1 DRY: client instantiation goes through one factory), `MockLoc.kt:127`.
- **Test:** injected shell that blocks forever — `stop()` still executes teardown after the timeout, and `busy` is observably released so a second `cast()` is accepted.
- **Blast radius if wrong:** medium. A too-short timeout cancels a legitimately slow cast on a cold head unit (the VD-discovery loop alone is 16×500 ms plus 8 s of `castSeq` sleeps). Start at 60 s, log every expiry, and never cancel `stop()` itself.

### WAVE 2 — wrong behaviour the owner has actually reported

**W2-1 · Teardown addressed at a service that may not exist; session identity does not survive the process** (RC-1: SAF-06 + ARCH-01) — **P1**
*Evidence: PROVEN.* `svcName` (:238) assigned only at `cast():374`; consumed at `stop():469`, `rollback():484-485`, `reconcileOnStart():836` — which never assigns it. `ClusterProfile.kt:60` defines `svcName="auto_container"` for DL5. Worse than the finding stated: `sc()` is also called at :334 and :338 inside cast()'s **warm branch, before** the :374 assignment — and `applyScaleLive:744` can set `casting=true` in a process that never ran `cast()`. So one press of a size button followed by an app switch skips the cluster-mode command entirely on DL5.
Corrections applied: scope is **DL5 + imported profiles only** (SEAL_DL3 and GENERIC_FALLBACK both use `"AutoContainer"`, so the entire proven fleet is unaffected). The silent paths are `reconcileOnStart`'s local `sh` (:800, drops stderr and exit code) and `rollback()` (:484, raw `adb.shell`, no logging at all) — `stop()`'s `sh` already surfaces stderr but then contradicts itself with "✅ Đã trả đồng hồ gốc".

- **Generic fix:** make the opcode command *unconstructible without a resolved profile*. Delete the `ClusterCast.svcName` field; move the builder to `fun ClusterProfile.svcCall(n: Int) = "service call $svcName 2 i32 1000 i32 $n s16 \"\""`. Every call site already holds a Context and already calls `ClusterProfile.resolve(app)`. Move the warm branch's two `sc()` calls behind the same resolve. Persist **the profile export string only**, one prefs key, written at :375 alongside `clusterMutated = true`, cleared in `stop()`'s `finally` — so a mid-session re-import cannot change the teardown sequence. Make `reconcileOnStart`'s and `rollback`'s `sh` read `errorOutput` and the exit code like `stop()`'s does.
- **Do NOT persist `vd`.** Opcode 16 recreates the VD with a new id (documented at :17, :340-342); `stop():434-438`, `applyScaleLive:725-726` and `reconcileOnStart:806` all correctly re-derive it. Persisting it would be a regression and directly contradicts CLAUDE.md §5.
- **Files:** `ClusterProfile.kt`, `ClusterCast.kt` (:238-239, :332-338, :374-375, :467-470, :484-485, :800, :836).
- **Test:** round-trip a DL5 profile; assert `profile.svcCall(18) == "service call auto_container …"`. Seam test: with a persisted DL5 profile and a fresh `ClusterCast` state, `reconcileOnStart`'s emitted teardown contains `auto_container`. Neither test can be written today — `sc()` is private and reads a global.
- **Blast radius if wrong:** low-medium. Deleting a field the compiler tracks is safe; the risk is the persisted profile going stale against a later import. Clear it in `finally`, never read it when absent.

Broaden `vdDirty` (:815-816) to enumerate the *same command set* `resetDisplayAll` undoes (CastShell.kt:180-183) — add `DisplayParse.density(dump,vd)` cur≠base and a new `DisplayParse.overscan(dump, vd)` parsing `mDisplayInfoOverscan=[l,t][r,b]`, already present in the dump the function fetches. **Rule to write into the code:** the dirty probe and the reset command list live adjacent so they cannot drift apart.

**W2-5 · Per-app frames are absolute pixels against a possibly-guessed reference** (RC-9: ARCH-09) — **P1**
*Evidence: PROVEN.* `AppScale` holds `rectL/T/R/B` as absolute px (:13-19); `boundsOn(w,h):24` returns them unscaled. `ClusterCastActivity.clusterRef():401-404` falls back to the profile's hardcoded 1920×720 when `lastClusterW/H` are unset — and those are only ever set inside `placeAppOnVd:517` / `applyScaleLive:746`, i.e. **after a successful cast**. ## 3. Conflicts and mandatory ordering

**C-1 (hard, safety) · W1-3 → W1-4.** The boot motion gate is `SpeedProvider.mpsOrNull()?.let { it < 0.5 } != true`. If `mpsOrNull` does not exist yet and the gate is written against `mps()`, it reads 0.0 on an unreadable HAL and **fails open in exactly the way W1-3 exists to close** — a motion gate that always says "stationary". Landing W1-4 first would ship a gate that is decorative.

**C-2 (hard, both directions) · W1-2 sweep vs. the DR state machine.** `DeadReckonModule.selfTest` flipping `active` false out from under a live session makes `MockLoc.push:57` silently no-op while `DeadReckonState.mocking` still reads true. If the sweep lands while `selfTest` still mutates, the sweep can observe that inconsistent pair and remove a provider that DR is about to re-arm. **Both halves of W1-2 must land in the same build**, never the sweep alone. Additionally make `push()` self-healing rather than silently inert.

**C-3 (hard) · W1-5 → W2-2.** The watchdog adds a periodic dadb consumer on the same single-thread `vdExec`. Today a hung cast already drains nothing; adding a 60 s watchdog behind an unbounded latch means the watchdog silently never runs — the exact failure it was built to prevent, invisibly. Timeout and cancellation first.

**C-4 (hard, and the trap in this plan) · W2-4 must NOT ship alone.** Today DiLink5 fails **closed**: `StackParse.parse` returns empty, `cast()` aborts at :283 before `castSeq` and before `placeAppOnVd`, so the cluster is never mutated. The two-character regex fix converts DL5 to **fail-open** — casts proceed, `wm overscan` hard-errors on A11+ (W3-4), and teardown addresses a non-existent service (W2-1). **W2-4 ships only in a build that also contains W2-1 and the `hasOverscan` gate from W3-4.** Until then, the correct short-term action for DL5 is to *improve the error message* at :283 (it currently says "mở app xong vẫn không thấy stack → HỦY", naming the wrong cause) and to fix the side effect that the abort leaves behind — the target app force-foregrounded on the driver's centre screen at :281.

**C-5 (direct conflict in a proposed API) · W3-1's `-1` sentinel vs. `restoreFullscreenOnMain`'s existing `vd` parameter.** Today that single parameter carries **two meanings**: it is (nominally) the scan scope, and it is the switch for `if (vd >= 1) resetDisplayAll` at CastShell.kt:94. Three call sites pass `-1` deliberately for the *second* meaning — "do not reset the cluster, it is still projecting" (ClusterCast.kt:328, :351; CastShell.kt:198). Redefining `-1` as "move nothing" silently changes those three sites' reset behaviour if the parameter is not split first. **Split into two explicit parameters (`fromVd`, `resetVd`) before touching the filter.** This conflict is not named in either source finding.

**C-6 (product conflict, needs a measurement not a decision) · W1-4 deleting `applyDefaultModes` vs. ISSUES.md:10-20 and item 3 ("xem lại có nên pre-seed ◈ cho AA không: nó đang chặn đúng cái rung duy nhất đưa AA lên được").** The team's own log says the ◈ flag was blocking R3 and that R3-force-stop — not the daemon — is probably what got Android Auto onto the cluster. Deleting the migration restores ◈ for AA and may make AA stop casting for owners who currently have it working. **This is CLAUDE.md §2 territory: mechanism vs. attribution, and the attribution is explicitly marked unproven.** Resolution: delete the migration (a migration must never widen permissions), and settle the AA question by the observation ISSUES.md already specifies — read the cast log panel during an AA cast and find the `R2 am display move-stack` line and whether `R3 ⚠ PHÁ PHIÊN` follows. Do not encode either answer as a default until that log exists.

**C-7 (data conflict) · W2-1 vs. W2-2 on what gets persisted.** SAF-05 asks to persist "the casting marker + vd + pkg"; ARCH-01 asks to persist a `ClusterSession(pkg, vd, profile, startedAt)`. **Both are wrong about `vd`** — opcode 16 recreates the VD with a new id, and three call sites already re-derive it correctly. Persist exactly two things: `pkg` and the profile export string. Everything else comes from `dumpsys`.

**C-8 (soft) · W2-5 (AppScale fractions) vs. W4-1 (SizingPlan).** Both rewrite the inputs to `applyBounds`. Land W2-5 first — it changes the data model, and rebasing a fresh `SizingPlan` onto changed units is cheaper than the reverse.

**C-9 (soft) · W2-8(b) anim marker vs. W1-4.** `MainActivity.kt:135` writes 0.5 on every open; `cast():379` reads it back. Whichever lands first must not leave a build where the marker exists but `cast()` still reads live — that would restore a stale marker over a value the user changed in between. One commit, both sides.

**Explicitly rejected as over-scoped:** ARCH-03's `SystemMutations` journal. For four of five mutations the prior value is either already persisted (PIP) or is the framework default recoverable by `reset`/`delete` (VD geometry, freeform flags). Only the animation scale has a genuine user-specific prior value, and W2-8(b) handles it in ~15 lines. Building the journal is a large refactor buying almost nothing.

## 4. On-car verification — before any build reaches the group

Do these **in order**, engine on, car **parked, handbrake on, in a place where you can see both screens**. Do not skip a step because the one before it looked fine. After every step, if anything looks wrong, stop and screenshot the "Chi tiết kỹ thuật" panel — do not try the next step.

1. **Write down the version.** Open ClusterNav. The title of the Chiếu screen must show a version number you have not tested before. If it shows a number you have already tested, stop — the build was not bumped (§9).
2. **Cold start.** Turn the car fully off, wait one minute, turn it on. Do not touch anything for three minutes. The instrument cluster must show the **normal stock gauge** the whole time — no map, no arrow, no black band, no flicker.
3. **Auto-cast, stationary.** If you have auto-cast turned on, it should project your chosen app about 25 seconds after start. Confirm it does, and confirm nothing on the centre screen was closed or restarted.
4. **Auto-cast, moving — the important one.** With a passenger driving, in a safe empty area, restart the head unit while rolling slowly (under 10 km/h). The cluster must **NOT** start projecting by itself. It must stay on the stock gauge.
5. **Cast (S1).** Press CHIẾU on your usual app. It appears on the cluster. The centre screen goes back to the launcher or the previous app — **not** a small floating window, not a shrunken screen.
6. **Size buttons (S2).** Press 100 / 90 / 80 / 70%. Each press must change the picture **on the cluster only**. Watch the centre screen carefully: it must not resize, not flicker, not turn into a window. Press ◀ ▲ ▼ ▶ and nhỏ/to. Then press 100% again and check the picture comes back exactly.
7. **The shape button.** Press ◠/▭. Either the cluster shape actually changes, or the button is not on screen at all. **A button that changes its own label and nothing else is a failure** — report it.
8. **Switch apps (S3).** Cast app A, then cast app B without pressing TẮT. B replaces A on the cluster. Time it: this must take about two seconds, not eight. The cluster must **not** flash black and it must not show the ADAS graphics again in between. App A must come back to the centre screen full-size.
9. **Stop (S4).** Press TẮT. Within a few seconds: the cluster returns to the stock gauge, the app is on the centre screen full-size, and the message says it returned the gauge. **Now check the launcher: press the home button and swipe through it.** If the launcher does not respond, that is the worst failure in this list — restart the head unit and report immediately.
10. **Stop while the app is dead (the watchdog).** Cast an app, then force-close that app from the head unit's own settings (do not press TẮT). Wait **two minutes**. The cluster must return to the stock gauge by itself. It must not stay projecting a frozen picture.
11. **Kill ClusterNav while casting.** Cast an app, then force-stop **ClusterNav** from settings. Open ClusterNav again. It must tidy up by itself: cluster back to the stock gauge, and the app back on the centre screen at the right size.
12. **After the previous two steps, open the "Chi tiết kỹ thuật" screen and screenshot it.** It should report the cluster as clean — no leftover size, no leftover density, no leftover PIP block, animations back to normal. If it reports anything left over, send the screenshot.
13. **The freeform undo button.** Find "Gỡ cờ freeform" in settings and press it. It should say it worked and tell you to switch the car off once. Do that at the end of the session and confirm the head unit starts up normally.
14. **Tunnel / GPS (S7).** Drive through a tunnel with a map app open. Position keeps moving sensibly, and **within 30 seconds of coming out** it snaps back to the real road. Afterwards, open any other map app and confirm it finds you correctly — **if any app on the car cannot find your position after this test, report it as the highest priority.**
15. **Tunnel entry from a cold start.** Turn the car off inside or right before a tunnel, turn it back on, drive through. The car must not show itself parked at your house. If it does, stop testing and report.
16. **The autotest is disarmed.** Without turning on the autotest switch in settings, ask a helper to try to launch it (or just confirm the switch is off). Then look at the cluster for three full minutes: there must be **no** "→ 250 m Nguyễn Huệ" or any turn instruction that you are not actually driving toward.
17. **Nav (S6/S8).** With the autotest OFF, start real navigation. The turn arrow, distance and street name on the cluster must match the phone/map. Take one wrong turn on purpose and confirm the cluster updates rather than freezing.
18. **Final sweep.** Park, press TẮT if anything is casting, open "Chi tiết kỹ thuật", screenshot. Then turn the car fully off and on and confirm the cluster comes up as the plain stock gauge.

**Any step that fails ⇒ the build does not go to the group.** Steps 4, 9, 10, 14, 15 and 16 are release blockers with no exception.

## 5. Verdict — incremental or rewrite?

**Incremental, with one deletion and three extractions. Do not rewrite `ClusterCast`.**

The reasoning, directly:

**What is actually wrong is smaller than it looks.** Six senior reviewers filed eight P0s; adversarial verification killed every one. Five of the eight failed on the same pattern — a correct AOSP mechanism attached to a driver-facing story that is unreachable on this hardware (SAF-02's launcher freeze needs a pinned stack on a `FLAG_OWN_CONTENT_ONLY` presentation display that never hosts one; SAF-03's floating window needs two mutually exclusive gates to both pass; ARCH-01's failure needs a DL5 head unit that appears nowhere in the field evidence). That is a **reporting discipline problem, not a code health problem**, and CLAUDE.md §2 already names it. The code itself is in better shape than the review round suggested: the ladder measures after every rung, `evictableOnVd` is correct and tested, PIP does marker-before-mutation properly, `applyBounds` and `applyScaleLive` both hard-guard `displayId != vd` at the execution layer exactly where §5 says the guard belongs. Someone was thinking clearly while writing this.

**What is genuinely broken is one idea repeated in eight places:** a value that lives only in RAM is used to decide something whose effects live on disk. `svcName`, `casting`, `freeformSeeded`, `savedAnim`, `lastGoodMps`, `active` in MockLoc, `busy`. Every one of them produces the same failure — a fresh process makes a confident decision about a world it has not looked at. That is fixable one field at a time, and each fix is small and independently testable. It does not need a rewrite; it needs the discipline that CLAUDE.md §5 already states applied consistently instead of five times out of eight.

**One module should be deleted, not rewritten: `T3Daemon.java` plus `escalateFreeform`.** 254 lines of Java with zero evidence it has ever placed an app, gates that are near mutually exclusive, a mutation path that acts on a target it re-derived by guessing, and a spec that already demoted it. The team's own issue log suspects the one field success attributed to it was actually rung R3. Keeping it means carrying RC-7 forever and maintaining the freeform seed that exists mainly to arm it. Delete it, and half of Wave 3 evaporates.

**Three extractions must happen, and they must happen in this order**, because each one removes a responsibility that makes the next one mechanical: the session record (W2-1) → the single move-stack gate (W3-1) → `SizingPlan` (W4-1). Only after all three should anyone split `ClusterCast.kt` (W4-2). Attempting the file split first — the tempting move, since it is the one thing a linter complains about — would shuffle 924 lines of untested logic across new files and destroy the only thing currently holding the behaviour stable, which is that one person can still read it end to end.

**The one thing I would change about how this code is being built, not what it contains:** `docs/specs/cluster-cast-v036.html:280` and `:290` both record SAF-11 as fixed, and it is not — the helper exists and `rollback()` calls a worse inline copy. A spec that records fixes that never got a call site is worse than no spec, because the next reviewer trusts it and skips the grep. Ship W2-7's one-line fix, correct both Reviewer Log entries, and add step 6 of CLAUDE.md §13 (`grep for a call site`) to the pre-commit routine as a hard gate rather than a habit.

**Ship order:** Wave 1 as one build, verified on the car through checklist steps 1-4 and 14-16 before anything else is merged. Wave 2 as two builds (W2-1/2/3 together; then W2-4 only alongside W3-4). Wave 3 and 4 unblocked after that, at whatever pace the maintainer can sustain — none of it is field-urgent, and pretending otherwise is how this repo got three different builds called v0.37.
